# 智能个人作品集系统 - 需求分析文档

## 1. 文档信息

项目名称：智能个人作品集系统
项目定位：面向秋招的高级个人作品展示、经历资产管理、工作模块复现、JD 智能适配与面试辅助系统
目标用户：本人、面试官、招聘方、后续维护者
当前阶段：需求分析阶段

## 2. 项目背景

传统个人主页通常只包含静态简历、项目列表和联系方式，展示维度较浅，难以证明候选人的真实工程能力。对于 Java 后端秋招场景，仅展示文字版项目经历不够有说服力，面试官更关注候选人是否真正理解业务链路、技术选型、问题排查、部署运维以及项目复盘。

本项目希望构建一个真实上线运行的前后端分离系统，将个人经历、实习经历、项目经历、技术能力、工作模块复现、JD 适配和面试准备整合到一个可扩展平台中。

项目不只是“个人主页”，而是一个：

- 高级个人作品集
- 在线简历系统
- 项目案例展示系统
- 工作模块复现实验室
- JD 智能适配器
- 面试准备工作台
- 后台内容管理系统
- DevOps 与 CI/CD 展示样板

## 3. 项目目标

### 3.1 业务目标

- 面向秋招面试，展示 Java 后端、微服务、Redis、MySQL、AI 应用接入、部署交付等能力。
- 将简历中的项目经历转化为可交互、可演示、可追问的工程案例。
- 支持根据不同岗位 JD 动态生成更匹配的个人介绍、项目排序和面试话术。
- 支持后续新增实习、项目、模块、文章和简历版本，减少硬编码维护成本。
- 实现真实上线和自动化部署，形成完整软件生命周期样板。

### 3.2 技术目标

- 采用前后端分离架构。
- 前端实现高级、克制、可扩展的个人作品集体验。
- 后端实现认证鉴权、内容管理、模块实验室、JD 适配、访问统计等能力。
- 使用数据库保存结构化经历资产。
- 使用 Redis 支持缓存、演示模块和高频数据场景。
- 接入 LLM 完成 JD 分析与内容生成。
- 使用 Docker、Nginx、Jenkins 实现自动化部署。
- 建立从需求、设计、开发、测试、部署到运维的完整工程流程。

## 4. 用户角色

### 4.1 访客

包括面试官、招聘方、同学、朋友等公开访问者。

权限：

- 访问首页、在线简历、项目案例、模块实验室、技术架构页。
- 查看公开项目和模块 Demo。
- 通过定制分享链接查看特定岗位版本。
- 不允许访问后台管理。

### 4.2 管理员

即本人。

权限：

- 登录后台。
- 管理个人信息、教育经历、实习经历、项目经历、技能、模块 Demo、文章。
- 管理 JD 适配结果和分享链接。
- 查看访问统计。
- 触发或查看部分系统配置。

### 4.3 面试模式访问者

面试过程中打开页面的一类特殊访问场景。

权限：

- 查看面试模式页。
- 查看推荐项目讲解顺序。
- 查看指定 Demo 和项目证据链。
- 访问内容应简洁、有重点，避免暴露后台或敏感数据。

## 5. 总体功能范围

系统分为以下功能域：

1. 公开展示站点
2. 在线简历与经历资产库
3. 项目案例系统
4. 工作模块复现实验室
5. JD 智能适配器
6. 面试模式
7. 后台管理系统
8. 访问统计与分享链接
9. 用户认证与权限控制
10. DevOps、CI/CD 与部署运维

## 6. 功能性需求

## 6.1 公开展示站点

### 6.1.1 首页

首页需要在短时间内展示个人定位、核心能力和重点项目。

需求：

- 展示姓名、求职方向、个人定位。
- 展示核心技术标签，例如 Java、Spring Boot、Spring Cloud、Redis、MySQL、AI 应用、Docker、Jenkins。
- 展示代表项目入口。
- 展示工作模块复现实验室入口。
- 展示 JD 智能适配器入口。
- 展示实习经历摘要。
- 展示 GitHub、邮箱、PDF 简历下载等联系入口。
- 支持读取默认内容或某个 `ProfileVariant` 内容。

验收标准：

- 首屏信息清晰，能在 10 秒内理解候选人方向。
- 项目和模块入口可点击。
- 在桌面端和移动端均正常展示。

### 6.1.2 在线简历页

需求：

- 展示教育背景。
- 展示专业技能。
- 展示实习经历。
- 展示项目经历。
- 展示竞赛、奖项、校内职务。
- 支持 PDF 简历下载。
- 手机号等敏感信息默认不直接公开展示，可通过 PDF 或受控方式提供。

验收标准：

- 内容与简历事实一致。
- 支持后续通过后台或数据源更新。
- 排版适合阅读和面试前快速浏览。

### 6.1.3 关于页 / 成长路线

需求：

- 展示从课程基础、项目实践、企业实习、AI 探索到个人作品集建设的成长路线。
- 可与经历、项目、技能关联。

验收标准：

- 时间线清楚。
- 能体现个人成长路径和学习能力。

## 6.3 项目案例系统

### 6.3.1 项目列表页

需求：

- 展示公开项目列表。
- 支持按技术栈、项目类型、时间筛选。
- 支持根据 JD Variant 调整项目排序。
- 每个项目卡片展示名称、摘要、技术栈、关键指标、关联 Demo。

### 6.3.2 项目详情页

需求：

- 展示项目背景、业务目标、技术栈、我的职责。
- 展示系统架构说明。
- 展示核心模块。
- 展示难点和方案。
- 展示量化成果。
- 展示项目证据链。
- 展示关联工作复现 Demo。
- 展示面试讲解点和可能追问。
- 支持 JD Variant 下的项目强调重点。

### 6.3.3 项目证据链

需求：

- 将业务问题、解决方案、技术实现、结果指标、关联 Demo、代码/接口、面试追问串联。

示例：

```text
业务问题：视频进度高频上报导致数据库写入压力
解决方案：Redis + 延迟任务
技术实现：播放中写 Redis，完播或延迟任务同步 MySQL
结果指标：重复写入减少约 70%，接口响应从 1.5s 降到 500ms
关联 Demo：视频播放学习链路复现
追问方向：缓存一致性、幂等、状态流转
```

验收标准：

- 每个重点项目至少有一条完整证据链。
- 证据链中的指标必须来自真实经历，不允许编造。

## 6.4 工作模块复现实验室

工作模块复现实验室用于复现简历中真实参与过的关键工作。

### 6.4.1 模块实验室首页

需求：

- 展示所有公开 Demo 模块。
- 支持按技术点筛选。
- 支持按项目关联筛选。
- 支持根据 JD Variant 推荐模块。

### 6.4.2 模块详情页统一结构

每个模块详情页应包含：

- 模块名称
- 来源项目
- 业务背景
- 我的参与内容
- 技术点
- 交互演示区
- 链路可视化区
- 日志与状态区
- 技术解释区
- 面试讲解点

### 6.4.3 可扩展模块机制

需求：

- 每个模块拥有 `ModuleDemo` 元数据。
- 前端通过 `demoType` 选择对应 Demo Renderer。
- 已有 Demo 类型可被复用。
- 新增模块时优先新增配置，必要时新增 Renderer。

基础 Demo 类型：

- `video-learning`
- `cache-sync`
- `state-machine`
- `agent-workflow`
- `rag-flow`
- `sql-bot`
- `api-flow`
- `file-parser`
- `alerting`
- `custom`

### 6.4.4 视频播放学习链路 Demo

需求：

- 模拟视频播放、暂停、拖拽、倍速。
- 定时上报播放进度。
- 展示前端请求参数。
- 展示后端处理链路。
- 展示 Redis 缓存状态。
- 展示 MySQL 最终状态。
- 展示学习状态变化。
- 支持模拟重复上报、网络失败、接口重试。

### 6.4.5 Redis 学习记录缓存同步 Demo

需求：

- 模拟高频写入。
- 展示 Redis 写入和 MySQL 同步差异。
- 展示重复写入减少效果。
- 展示响应时间对比。
- 支持手动触发延迟同步。

### 6.4.6 Agent 工作流 Demo

需求：

- 展示用户输入。
- 展示意图识别。
- 展示工具路由。
- 展示 SQL Bot、RAG、LLM 回答等节点。
- 展示节点输入、输出、耗时、状态。
- 支持模拟失败和重试。

验收标准：

- 第一阶段至少完成一个高质量 Demo。
- Demo 不只是静态展示，必须有真实交互和状态变化。
- Demo 与项目经历和证据链关联。

## 6.5 JD 智能适配器

### 6.5.1 JD 输入与分析

需求：

- 用户可粘贴岗位 JD。
- 系统调用后端接口进行分析。
- 后端调用 LLM，结合个人经历资产生成结构化结果。

### 6.5.2 分析结果

结果应包含：

- 岗位方向
- 技术关键词
- 必备要求
- 加分项
- 匹配度
- 高匹配能力
- 中匹配能力
- 低匹配或风险项
- 推荐项目排序
- 推荐 Demo 模块
- 岗位适配版个人介绍
- 面试自我介绍
- 项目强调重点
- 面试追问建议

### 6.5.3 真实性约束

需求：

- LLM 只能基于已有经历、项目、技能和模块数据生成。
- 不允许生成不存在的公司、项目、指标、技术经历。
- 对缺少支撑的 JD 要求必须输出风险提示。
- 每个重点生成内容应附带 `supportedBy` 来源。

### 6.5.4 ProfileVariant

需求：

- 用户可保存某次 JD 分析结果为 Profile Variant。
- Variant 可用于首页、项目页、面试模式和分享链接。

验收标准：

- 输入 JD 后能返回结构化结果。
- 生成结果可保存。
- 页面能读取 Variant 展示不同内容。

## 6.6 面试模式

需求：

- 根据默认内容或某个 JD Variant 生成面试导览。
- 展示 30 秒、1 分钟、3 分钟自我介绍。
- 展示推荐项目讲解顺序。
- 展示推荐打开的 Demo。
- 展示每个项目的必讲点。
- 展示可能追问和回答要点。
- 支持一键进入相关项目或 Demo 页面。

验收标准：

- 面试前可快速打开使用。
- 内容简洁、有重点。
- 不暴露后台和敏感配置。

## 6.7 定制分享链接

需求：

- 管理员可基于 ProfileVariant 创建分享链接。
- 分享链接可设置名称、slug、启用状态、过期时间。
- 访问分享链接后展示对应 Variant 内容。
- 可记录基础访问数据。

验收标准：

- 不同分享链接展示不同重点。
- 禁用或过期链接无法访问。

需求：

验收标准：

## 7. 非功能性需求

## 7.1 性能需求

- 首页首屏加载应尽量控制在 2 秒内。
- 静态资源应进行压缩和缓存。
- 首页、项目页等公开内容可做服务端渲染或静态生成。
- 高频 Demo 数据和热点内容可使用 Redis 缓存。
- JD 分析属于耗时操作，应提供 loading、超时和失败提示。

## 7.2 可用性需求

- 桌面端体验优先，同时适配移动端。
- 页面结构清晰，避免过度动效影响阅读。
- 所有交互按钮应有明确反馈。
- Demo 页面应提供重置能力。
- 错误状态、空状态、加载状态必须设计。

## 7.3 可维护性需求

- 经历、项目、技能、模块数据结构化。
- 页面组件与内容数据解耦。
- 前端组件分层清晰。
- 后端采用统一响应体、统一异常处理、参数校验。
- 关键业务逻辑需要注释和文档。
- 文档随功能迭代更新。

## 7.4 可扩展性需求

- 支持新增项目和经历。
- 支持新增 ModuleDemo。
- 支持新增 DemoRenderer。
- 支持新增 LLM Provider。
- 支持新增分享链接和简历版本。
- 支持后续 PDF 导出、更多访问统计和复杂 Agent 编排。

## 7.5 安全需求

- 后台必须登录后访问。
- 管理接口必须鉴权。
- LLM API Key、数据库密码等敏感配置不得提交到仓库。
- 提供 `.env.example`。
- 公开 Demo 接口要限制写入范围。
- JD 输入要做长度限制。
- 防止 XSS，用户输入内容展示前需安全处理。
- 后台操作需要基本审计日志。

## 7.6 可靠性需求

- 后端接口错误应返回统一格式。
- LLM 调用失败时应有降级提示。
- Jenkins 部署失败时不应破坏上一版可用服务。
- Docker 服务应支持健康检查。
- 关键服务需要日志记录。

## 7.7 兼容性需求

- 支持现代 Chrome、Edge、Safari。
- 移动端支持主流浏览器访问。
- 前端布局应支持不同屏幕宽度。

## 8. 数据需求

核心实体：

- User
- Profile
- Education
- Experience
- Project
- Skill
- ModuleDemo
- EvidenceChain
- Article
- JDAnalysis
- ProfileVariant
- ShareLink
- VisitEvent

实体关系简述：

- Profile 拥有多个 Education、Experience、Project、Skill。

- VisitEvent 可关联页面、项目、Demo、ShareLink。

## 9. 外部接口需求

### 9.1 LLM Provider

用途：

- JD 分析
- 个人介绍生成

要求：

- 后端需要抽象 Provider 接口。
- 支持配置模型、API Key、Base URL。
- 返回结构化 JSON。
- 保存 Prompt 版本或至少在代码中集中管理 Prompt。

用途：

- 后续可用于联系表单、告警 Demo、系统通知。

第一期可选。

### 9.3 GitHub / CI Webhook

用途：

- GitHub push 触发 Jenkins 构建。

## 10. CI/CD 与部署需求

### 10.1 基础部署形态

推荐部署结构：

```text
Nginx
  ├─ /           -> Next.js 前端
  └─ /api        -> Spring Boot 后端

Spring Boot
  ├─ MySQL
  ├─ Redis
```

## 14. 风险分析

### 14.1 范围过大风险

风险：

- 功能较多，容易开发周期过长。

应对：

- 坚持 P0/P1/P2 分期。
- 第一阶段先完成可上线闭环。

### 14.2 前端高级感不足风险

风险：

- 页面容易变成普通模板。

应对：

- 参考高质量开源 portfolio 和 dashboard。
- 先做真实页面，再通过浏览器截图逐步打磨。
- 控制配色、间距、排版和动效。

### 14.3 LLM 编造内容风险

风险：

- 生成内容可能夸大或编造经历。

应对：

- 使用结构化经历数据作为上下文。
- Prompt 明确禁止编造。
- 输出 `supportedBy` 和 `riskNotes`。
- 保存前提供人工确认。

### 14.4 部署复杂度风险

风险：

- Jenkins、Docker、Nginx、服务器部署可能占用较多时间。

应对：

- 先实现 Docker Compose。
- Jenkins 放到 P1。
- 编写清晰部署文档。

### 14.5 数据安全风险

风险：

- 简历中包含手机号、邮箱等敏感信息。

应对：

- 公开页面隐藏或弱化手机号。
- PDF 简历可单独控制。
- 后台和环境变量严格保护。

## 15. 验收标准

第一阶段完成标准：

- 本地可以启动前端和后端。
- 首页、简历页、项目页可访问。
- 经历和项目内容来自结构化数据。
- 至少一个工作模块 Demo 可交互。
- JD 输入后能生成结构化适配结果。
- 可通过分享链接展示不同内容版本。
- Docker Compose 可启动主要服务。

最终上线标准：

- 网站可通过域名访问。
- 前后端服务稳定运行。
- 后台可维护内容。
- Jenkins 可自动部署。
- 关键页面在桌面和移动端显示正常。
- 项目文档完整，能支撑面试讲解。
